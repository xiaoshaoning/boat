#### 2. 图遍历公共API完善
- **缺失函数添加**：实现`boat_graph_get_edge_at_index()`公共函数
- **API一致性**：与现有的`boat_graph_get_node_at_index()`保持相同设计模式
- **安全访问**：提供边界检查，无效索引返回NULL
- **实现位置**：在`graph.c`中添加函数定义

```c
// graph.h声明
BOAT_API boat_edge_t* boat_graph_get_edge_at_index(const boat_graph_t* graph, size_t index);

// graph.c实现
boat_edge_t* boat_graph_get_edge_at_index(const boat_graph_t* graph, size_t index) {
    if (!graph || index >= graph->edge_count) {
        return NULL;
    }
    return graph->edges[index];
}
```

#### 3. 模型模块私有依赖消除
- **头文件清理**：从`model.c`中移除`#include "../graph/graph_private.h"`
- **API迁移**：将所有直接访问`graph->edges[]`替换为`boat_graph_get_edge_at_index()`
- **计数函数替换**：将`graph->edge_count`替换为`boat_graph_edge_count(graph)`
- **封装增强**：完全使用公共API访问图结构，消除模块间耦合

```c
// 重构前（直接访问内部结构）
for (size_t j = 0; j < graph->edge_count; j++) {
    boat_edge_t* edge = graph->edges[j];
    // ...
}

// 重构后（使用公共API）
for (size_t j = 0; j < boat_graph_edge_count(graph); j++) {
    boat_edge_t* edge = boat_graph_get_edge_at_index(graph, j);
    // ...
}
```

### 🔧 技术实现细节

#### 1. 符号导出修复策略
```
问题根源：
- Windows DLL需要显式导出符号（__declspec(dllexport/dllimport)）
- graph.h函数缺少BOAT_API宏，导致测试程序链接时符号不可见

解决方案：
1. 确保graph.h包含export.h（提供BOAT_API宏定义）
2. 为所有公共函数添加BOAT_API前缀
3. 保持与tensor.h、model.h等模块的一致性

技术影响：
- Windows: 生成正确的DLL导入/导出声明
- Linux/macOS: 使用visibility("default")属性控制符号可见性
- 跨平台兼容：单一代码库支持所有平台
```

#### 2. 公共API设计原则
```
设计目标：
1. 完整性：提供所有必要的图遍历和访问接口
2. 安全性：边界检查防止无效访问
3. 一致性：与现有API保持相同模式和命名约定
4. 封装性：隐藏内部数据结构，仅通过函数访问

API扩展模式：
- 索引访问：get_node_at_index, get_edge_at_index
- 计数查询：node_count, edge_count, in_degree, out_degree
- 遍历支持：dfs, bfs, topological_sort
- 属性访问：node_data, node_type, edge_source, edge_target
```

#### 3. 模块解耦架构
```
原始架构问题：
- model.c直接包含graph_private.h，访问图内部结构
- 模块间紧耦合，违反封装原则
- 图结构变更会影响模型模块

重构后架构：
- model.c仅包含公共graph.h头文件
- 通过boat_graph_get_edge_at_index等函数访问边
- 通过boat_graph_edge_count等函数获取图状态
- 模块边界清晰，依赖关系明确

架构优势：
1. 封装性：图内部结构变化不影响模型模块
2. 可维护性：各模块可独立开发和测试
3. 可测试性：可模拟图API进行单元测试
4. 可扩展性：易于添加新的图访问函数
```
