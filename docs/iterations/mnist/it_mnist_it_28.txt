**修改文件**: `examples/mnist/mnist_autodiff.c`
**修改范围**: 调试系统重构
**技术细节**:

#### 3.1 运行时调试级别变量
```c
// 移除强制DEBUG_LEVEL定义（原第7-8行）
// 添加运行时调试级别变量
static int g_debug_level = DEBUG_LEVEL;
```

#### 3.2 重构调试打印宏
```c
// 将编译时条件宏改为运行时检查
#define DEBUG_PRINT(fmt, ...) do { \
    if (g_debug_level >= 4) { \
        fprintf(stderr, "DEBUG: " fmt, ##__VA_ARGS__); \
    } \
} while(0)

// 类似地重构INFO_PRINT、WARN_PRINT、ERROR_PRINT宏
```

#### 3.3 添加命令行参数支持
```c
// 在参数解析循环中添加
else if (strcmp(argv[i], "--debug-level") == 0 && i + 1 < argc) {
    g_debug_level = atoi(argv[i + 1]);
    i++; // Skip next argument
    printf("Debug level set to %d\n", g_debug_level);
}

// 更新帮助信息
printf("Usage: %s [--load <checkpoint_file>] [--debug-level <level>]\n", argv[0]);
printf("       --load <file>        Load model weights from checkpoint file\n");
printf("       --debug-level <n>    Set debug output level (0-4, default=0)\n");
```

**影响**: 用户现在可以通过命令行参数动态控制调试输出级别，无需重新编译。

## 📝 技术决策记录

### 决策1: 运行时调试级别 vs 编译时
**问题**: 调试级别应该在编译时固定还是运行时可调？
**决策**: 采用运行时变量控制
**依据**: 提供更大灵活性，用户无需重新编译即可调整调试级别
**影响**: 轻微运行时开销，但用户体验大幅改善

### 决策2: 全局变量 vs 参数传递
**问题**: 如何传递调试级别到各个函数？
**决策**: 使用静态全局变量 `g_debug_level`
**依据**: 简单有效，避免函数签名修改
**影响**: 模块间耦合增加，但调试系统本就是全局性功能

### 决策3: 保持进度输出直接刷新
**问题**: 是否所有printf都应添加fflush？
**决策**: 仅在关键进度输出后添加fflush
**依据**: 平衡性能和用户体验，关键进度信息需要及时显示
**影响**: 确保用户始终能看到最新训练进展
