### Priority 2: Enhanced CMake Configuration (COMPLETE)

**Updated CMakeLists.txt Features**:

1. **Flexible PyTorch Path Configuration**:
```cmake
# Support both standard LibTorch installations and Python-extracted libraries
if(BOAT_PYTORCH_PATH)
    message(STATUS "Using PyTorch from: ${BOAT_PYTORCH_PATH}")

    if(EXISTS "${BOAT_PYTORCH_PATH}/share/cmake/Torch/TorchConfig.cmake")
        # Standard LibTorch installation
        set(Torch_DIR "${BOAT_PYTORCH_PATH}/share/cmake/Torch")
        find_package(Torch REQUIRED)
    else()
        # Manual configuration (e.g., from Python installation)
        message(STATUS "Manual PyTorch configuration from Python installation")

        # Set include directories
        set(TORCH_INCLUDE_DIRS
            "${BOAT_PYTORCH_PATH}/include"
            "${BOAT_PYTORCH_PATH}/include/torch/csrc/api/include"
        )

        # Find required libraries manually
        set(TORCH_REQUIRED_LIBS torch torch_cpu c10 fmt cpuinfo sleef)
        set(TORCH_LIBRARIES "")

        foreach(lib ${TORCH_REQUIRED_LIBS})
            find_library(TORCH_${lib}_LIBRARY
                NAMES ${lib}
                PATHS "${TORCH_LIBRARY_DIR}"
                NO_DEFAULT_PATH
            )
            if(TORCH_${lib}_LIBRARY)
                list(APPEND TORCH_LIBRARIES ${TORCH_${lib}_LIBRARY})
            endif()
        endforeach()
    endif()
endif()
```

2. **Dynamic Library Linking**:
```cmake
if(BOAT_WITH_PYTORCH)
    if(DEFINED TORCH_LIBRARIES AND TORCH_LIBRARIES)
        # Use TORCH_LIBRARIES variable set by configuration
        target_link_libraries(boat PUBLIC ${TORCH_LIBRARIES})
        message(STATUS "Linking with Torch libraries: ${TORCH_LIBRARIES}")
    else()
        # Fallback: manually specify required libraries
        target_link_libraries(boat PUBLIC
            torch
            torch_cpu
            c10
            fmt
            cpuinfo
            sleef
        )
    endif()
endif()
```

Technical Challenges and Solutions
----------------------------------

### Challenge 1: Architecture Mismatch (ARM64 vs x64)

**Problem**:
- PyTorch nightly downloads for Windows (`libtorch-win-shared-with-deps-latest.zip`) contain ARM64 libraries
- Linker errors: "无法解析的外部符号" (unresolved external symbols) due to architecture mismatch
- 139 unresolved external commands when linking

**Solution**:
1. **Investigated PyTorch distribution channels**:
   - Official downloads: Default to ARM64 for Windows nightly builds
   - GitHub releases: Limited Windows x64 availability
   - Python installation: Contains x64 libraries for current platform

2. **Extraction from Python installation**:
   - PyTorch Python package (`torch==2.10.0+cu130`) contains x64 libraries
   - Libraries located in `site-packages/torch/lib/` directory
   - Complete header set in `site-packages/torch/include/`

3. **Library verification**:
   - Confirmed x64 architecture through file sizes and linking tests
   - No ARM64 warnings during build process
   - All required dependencies available

### Challenge 2: CMake Configuration Flexibility

**Problem**:
- Original CMake assumed standard LibTorch installation structure
- Required `share/cmake/Torch/TorchConfig.cmake` file
- Python-extracted libraries lack CMake configuration files

**Solution**:
1. **Enhanced detection logic**:
   - Check for standard LibTorch installation first
   - Fall back to manual configuration if CMake files not found
   - Support both installation types through single configuration path

2. **Manual library detection**:
   - Implement `find_library()` for each required library
   - Set include directories manually
   - Build `TORCH_LIBRARIES` list dynamically

3. **Conditional linking**:
   - Use `TORCH_LIBRARIES` if available from configuration
   - Fall back to hardcoded library list as backup
   - Provide clear logging for debugging

### Challenge 3: C/C++ Language Interoperability

**Problem**:
- Boat framework primarily C language
- PyTorch/LibTorch is C++ library
- Need clean C API for framework users

**Solution**:
1. **C++ wrapper implementation**:
   - Single C++ source file (`pytorch.cpp`) handles PyTorch interaction
   - C-compatible functions with `extern "C"` linkage
   - Clean separation between C and C++ code

2. **Memory management bridging**:
   - Convert between PyTorch tensor memory and Boat tensor memory
   - Reference counting for shared resources
   - Proper cleanup on both sides

3. **Error handling translation**:
   - Catch C++ exceptions and convert to C error codes
   - Consistent error reporting across language boundary
   - Memory safety during error conditions

### Challenge 4: Windows-specific Build Issues

**Problem**:
- MSVC compiler requires specific flags for C++17
- Windows linker needs proper runtime library configuration
- DLL export/import semantics

**Solution**:
1. **Compiler flags**:
   - `/std:c++17` for C++ standard compliance
   - `/EHsc` for exception handling
   - Warning level adjustments for third-party code

2. **Linker configuration**:
   - `/NODEFAULTLIB:libcmt.lib` to avoid runtime library conflicts
   - Proper DLL export definitions
   - Import library generation

3. **Runtime compatibility**:
   - Ensure all required DLLs are available at runtime
   - Path configuration for development vs deployment
   - Dependency version matching
