### 🔧 技术实施与修改

#### 1. 调用约定宏优化

**修改文件**: `include/boat/export.h`
```c
// Calling convention macros
#if BOAT_WINDOWS && defined(_M_X64)
    // Windows x64: default calling convention (no explicit specifier needed)
    #define BOAT_CALL
#elif BOAT_WINDOWS
    // Windows x86: use __stdcall for compatibility
    #define BOAT_CALL __stdcall
#else
    // Non-Windows platforms
    #define BOAT_CALL
#endif

// Compiler-specific export/import macros (不包含 BOAT_CALL，避免编译错误)
#if BOAT_WINDOWS
    #ifdef BOAT_BUILDING_DLL
        #define BOAT_API __declspec(dllexport)
    #else
        #define BOAT_API __declspec(dllimport)
    #endif
#else
    #if __GNUC__ >= 4
        #define BOAT_API __attribute__((visibility("default")))
    #else
        #define BOAT_API
    #endif
#endif
```

**关键修改**:
- 区分 Windows x64 和 x86 的调用约定
- x64 使用默认调用约定（无显式说明符）
- x86 使用 `__stdcall` 确保兼容性
- BOAT_API 不再自动包含 BOAT_CALL，避免与函数声明语法冲突

### 🔍 问题分析

#### 1. 当前问题特征
- **选择性失败**: 梯度检查测试通过，但层接口失败
- **函数地址有效**: 表明动态链接成功，符号正确解析
- **无调试输出**: 添加到函数的调试代码未执行，表明函数可能未到达预期代码路径
- **平台特异性**: 仅 Windows 出现，Linux/macOS 预期正常

#### 2. 可能原因
1. **调用约定仍未完全匹配**: 虽然添加了 BOAT_CALL，但 Windows x64 默认调用约定可能仍需特殊处理
2. **函数指针类型不匹配**: 测试中通过函数指针调用可能引入额外问题
3. **DLL 边界问题**: Windows DLL 导出/导入可能仍有名称修饰或调用约定问题
4. **堆栈对齐问题**: x64 Windows 要求的堆栈对齐可能未满足

#### 3. 排除的假设
- ✅ **函数实现错误**: 梯度检查测试通过，证明算法正确
- ✅ **内存管理问题**: 无内存泄漏，引用计数正常
- ✅ **基本调用约定**: 已添加平台特定调用约定宏
