#### 4. 预防性代码审查
- **目标**: 识别和修复其他潜在的平台兼容性问题
- **审查范围**:
  1. **所有 DLL 导出函数**: 检查是否存在简单包装函数
  2. **层接口函数**: 特别是 `*_layer_backward` 和 `*_layer_forward` 包装器
  3. **工具函数**: 简单辅助函数可能被编译器优化消除
  4. **初始化/清理函数**: 简单构造函数和析构函数
- **审查标准**:
  - 函数体是否简单（少于 10 行代码）
  - 是否直接返回参数或常量值
  - 是否缺乏副作用（无内存分配、无外部调用）
  - 是否在多个编译单元中使用
- **修复策略**:
  - 添加 `__declspec(noinline)` 到简单包装函数
  - 使用 `BOAT_NOINLINE` 宏统一处理跨平台
  - 考虑函数合并或重构，避免不必要的包装

#### 5. 跨平台兼容性优化
- **目标**: 建立统一的跨平台调用约定基础设施
- **实施步骤**:
  1. 定义 `BOAT_NOINLINE` 宏，统一处理各平台 noinline 属性
  2. 完善 `BOAT_CALL` 宏，确保所有平台调用约定一致性
  3. 添加平台检测和条件编译基础设施
  4. 创建平台兼容性测试套件
  5. 验证 Linux/macOS 平台未受 Windows 修复影响
- **宏定义提案**:
  ```c
  // include/boat/export.h
  #ifdef _MSC_VER
    #define BOAT_NOINLINE __declspec(noinline)
  #elif defined(__GNUC__) || defined(__clang__)
    #define BOAT_NOINLINE __attribute__((noinline))
  #else
    #define BOAT_NOINLINE
  #endif
  ```

#### 6. 构建系统改进
- **目标**: 优化 DLL 构建配置，预防编译器和链接器优化问题
- **实施步骤**:
  1. 检查 CMake 配置中的编译器优化选项
  2. 添加链接器选项控制（如 `/OPT:REF`、`/OPT:ICF`）
  3. 创建专门的 DLL 构建配置（DebugDLL、ReleaseDLL）
  4. 添加构建时导出符号验证步骤
  5. 优化 PDB 文件生成，增强调试能力
- **关键配置**:
  - Windows MSVC: `/Gy-` 禁用函数级链接，`/OPT:NOREF` 防止未引用函数消除
  - GCC/Clang: `-fno-inline-functions`、`-fno-inline-small-functions`
